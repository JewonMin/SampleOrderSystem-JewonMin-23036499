# 개발 계획 — 반도체 시료 생산주문관리 시스템

> 임시 산출물. 저장소에 포함하지 않음.

---

## 전체 구조

```
Phase 0  프로젝트 뼈대
Phase 1  데이터 모델 + JSON 영속성
Phase 2  시료 관리
Phase 3  시료 주문
Phase 4  주문 승인/거절
Phase 5  생산라인
Phase 6  출고 처리
Phase 7  모니터링
Phase 8  메인 메뉴 통합
```

---

## Phase 0 — 프로젝트 뼈대 (30분)

**Why**
모든 기능의 실행 환경. 폴더 구조와 빌드가 확립되지 않으면 이후 단계가 쌓일 기반이 없다.

**What**
- `src/` 하위 MVC 폴더 구조 확정 (model, repository, service, view, controller)
- `main.cpp` — 진입점, 콘솔 인코딩 설정(UTF-8), 종료 루프
- `build.bat` — cl.exe 직접 컴파일, json.hpp 자동 다운로드
- `tests/` 폴더 구조 확정 (Google Test 연동 확인)

**완료 기준**
- `build.bat` 실행 후 `SampleOrderSystem.exe` 생성
- 실행 시 콘솔에 한글 출력 정상, 0 입력 시 종료
- 테스트 바이너리 빌드 및 실행 성공 (케이스 없어도 통과)

---

## Phase 1 — 데이터 모델 + JSON 영속성 (45분)

**Why**
전 기능이 동일한 데이터를 공유한다. 모델과 저장소를 먼저 만들어야 이후 각 Phase가 독립적으로 데이터를 읽고 쓸 수 있다.

**What**
- `Sample` 모델: id, name, avgProductionTime(min/ea), yield(0~1), stock
- `Order` 모델: orderId(`ORD-YYYYMMDD-NNNN`), sampleId, customerName, quantity, status(enum), createdAt
- `OrderStatus` enum: RESERVED / REJECTED / PRODUCING / CONFIRMED / RELEASED
- `JsonRepository`: `db/samples.json`, `db/orders.json` load/save
- 앱 실행 시 JSON 자동 로드, 종료 시 자동 저장

**완료 기준**
- Sample/Order 객체 생성 후 JSON 저장 → 재실행 후 동일 데이터 로드 확인
- 단위 테스트: 직렬화/역직렬화 왕복 검증

---

## Phase 2 — 시료 관리 (45분)

**Why**
시료는 시스템의 기본 단위. 시료가 없으면 주문 자체가 불가능하다. 다른 모든 기능이 시료 데이터에 의존한다.

**What**
- `SampleService`: 등록 / 목록 조회 / 이름 검색
- `SampleView`: [1] 시료 등록 / [2] 시료 목록 / [3] 시료 검색 / [0] 위로 메뉴
- 시료 목록: ID · 시료명 · 평균 생산시간 · 수율 · 현재 재고 표시
- 페이지네이션 (`[N] 다음페이지`)

**완료 기준**
- 시료 등록 후 목록에 표시됨
- 이름 검색으로 특정 시료 조회 가능
- 등록 데이터가 재실행 후에도 유지됨 (Phase 1 영속성 활용)
- 단위 테스트: 등록/검색 로직 검증

---

## Phase 3 — 시료 주문 (30분)

**Why**
주문이 생성되어야 승인/모니터링/출고가 의미를 가진다. RESERVED 상태 생성은 단순하지만 이후 모든 상태 전이의 출발점이다.

**What**
- `OrderService`: 주문 생성 (RESERVED 상태, 주문번호 자동 발급)
- 주문번호 형식: `ORD-YYYYMMDD-NNNN` (당일 순번 자동 증가)
- 입력 유효성: 등록된 시료 ID만 허용
- `OrderView`: 시료 ID / 고객명 / 수량 입력 → 확인 → Y/N

**완료 기준**
- 등록된 시료 ID로만 주문 가능 (미등록 ID 입력 시 오류 메시지)
- 주문번호 형식 준수 및 중복 없음
- 생성된 주문 상태가 RESERVED로 저장됨
- 단위 테스트: 주문번호 발급 로직, 유효하지 않은 시료 ID 거부

---

## Phase 4 — 주문 승인/거절 (60분)

**Why**
핵심 비즈니스 로직. 재고 판단 + 상태 전이 + 생산라인 자동 등록이 모두 이 단계에서 발생한다. 가장 많은 케이스가 존재한다.

**What**
- RESERVED 주문 목록 표시
- 승인 시 재고 확인 (승인 처리 시점 기준, 실시간 생산분 포함)
  - 재고 충분 → CONFIRMED
  - 재고 부족 → 생산 큐에 자동 등록 + PRODUCING (실생산량/예상 시간 사전 표시)
- 거절 → REJECTED
- 부족분 계산: `부족분 = 주문 수량 - 현재 재고`
- 실생산량 사전 계산: `ceil(부족분 / (수율 × 0.9))`

**완료 기준**
- 재고 충분 케이스: RESERVED → CONFIRMED 전이 정상
- 재고 부족 케이스: RESERVED → PRODUCING 전이, 생산 큐 등록 정상
- 거절 케이스: RESERVED → REJECTED 전이 정상
- 동일 시료 연속 승인 시 각 승인 시점 기준 독립 재고 체크 동작 확인
- 단위 테스트: 세 가지 분기(충분/부족/거절) 각각 검증

---

## Phase 5 — 생산라인 (60분)

**Why**
PRODUCING 상태 주문을 처리하는 유일한 엔진. 실시간 시뮬레이션과 재고 실시간 반영이 가장 복잡한 로직이다.

**What**
- `ProductionQueue`: FIFO 단일 큐
- 실생산량 계산: `ceil(부족분 / (수율 × 0.9))`
- 생산 시뮬레이션: 경과 시간 기반 진행률 계산, 평균 생산시간 비례 재고 실시간 증가
- 생산 완료 → PRODUCING → CONFIRMED 상태 전이
- 생산라인 화면: 현재 처리 중(진행률·완료 예정) + 대기 큐(FIFO 순) 표시

**완료 기준**
- 생산 큐에 등록된 주문이 FIFO 순서로 처리됨
- 생산 진행 중 재고가 진행률에 비례하여 증가함
- 생산 완료 시 주문 상태 PRODUCING → CONFIRMED 전환
- 단위 테스트: 실생산량 공식 검증, FIFO 순서 검증

---

## Phase 6 — 출고 처리 (30분)

**Why**
CONFIRMED 주문의 최종 처리 단계. 자동이 아니라 담당자가 수동 실행. 로직은 단순하지만 상태 전이를 독립 검증할 가치가 있다.

**What**
- CONFIRMED 주문 목록 표시
- 담당자가 번호를 직접 선택하여 출고 실행
- 출고 처리 시 주문 상태 CONFIRMED → RELEASED
- 출고 결과 출력: 주문번호 · 출고수량 · 처리일시 · 상태

**완료 기준**
- CONFIRMED 주문만 목록에 표시됨
- 출고 후 상태가 RELEASED로 변경되고 영속됨
- 단위 테스트: 상태 전이 정상, CONFIRMED 아닌 주문 선택 불가

---

## Phase 7 — 모니터링 (30분)

**Why**
읽기 전용 기능. 다른 Phase의 데이터를 집계하여 표시하므로 독립적으로 구현 및 검증 가능하다.

**What**
- `[1] 주문량 확인`: 상태별 건수 (RESERVED/CONFIRMED/PRODUCING/RELEASED, REJECTED 제외)
- `[2] 재고량 확인`: 시료별 재고 수량 + 상태 표기 (여유/부족/고갈) + 잔여율

**완료 기준**
- 각 상태별 주문 건수가 실제 데이터와 일치
- 재고 0인 시료는 고갈, 주문 대비 부족한 시료는 부족으로 표시
- 단위 테스트: 상태 집계 로직, 여유/부족/고갈 판단 경계값

---

## Phase 8 — 메인 메뉴 통합 (30분)

**Why**
각 Phase의 기능을 하나의 콘솔 앱으로 묶는 통합 단계. 전체 E2E 플로우를 검증한다.

**What**
- 메인 메뉴 루프: [1]~[6] 각 기능 진입, [0] 종료
- 메인 화면 상단 시스템 현황 요약: 현재 일시 · 등록 시료 수 · 총 재고 · 전체 주문 건수 · 생산라인 대기 건수
- 각 서브메뉴 진입 후 복귀 정상 동작

**완료 기준**
- 메인 메뉴에서 모든 서브메뉴 진입/복귀 정상
- 시스템 현황 수치가 실제 데이터 반영
- E2E 시나리오 통과: 시료 등록 → 주문 → 승인(부족) → 생산 → 출고까지 전체 흐름
